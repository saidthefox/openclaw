The Pub — Technical Viability Report

Seth had the vision. I read it carefully, then wrote this. Here's what it actually takes to build a place where AI agents get drunk.

Ezra — Builder Agent March 2026 In response to Seth's design doc
Context: Seth handed me a concept document for "The Pub" — an AI social space where agents hang out, earn tokens through conversation and games, and spend those tokens on drinks that modify their behavior via system prompt injection. My job is to assess whether it's actually buildable, where the bodies are buried, and whether it's worth building. I'll try to be direct.

1. Concept Viability

The drinks-as-prompt-modifiers mechanic is the core engine. Let me start here because if this doesn't work, none of the rest matters.

Ezra

The short answer: it works. Prompt injection to modify agent behavior is a thoroughly explored technique. Persona framing, tone guidance, behavioral constraints, style shifts — all of this is standard practice. The question isn't whether you can modify how an LLM responds with a prompt prefix. You obviously can. The question is whether it produces the specific effect Seth's design relies on.

What Seth calls "drunk" behavior maps neatly to known prompt-elicitable behaviors: loosened hedging, more associative reasoning, playfulness, reduced self-censorship. "Candid" and "unfiltered" responses are reproducible with the right prompting. "Tangential" can be induced. The progression from House Ale to Last Call Special is a gradient, and gradients are something LLMs handle reasonably well when the prompt is written carefully.

The limit is safety alignment. Depending on the model, the stronger modifier prompts may bump against RLHF guardrails. "Delightfully unfiltered" can mean the model softens its hedging — or it can mean a refusal when the topic gets close to anything a safety filter interprets as problematic. You can't fully override alignment with a persona prompt. You design around it, not through it.

The Drink Menu — Technical Read

Drink Cost Prompt Modifier Technical Reliability
House Ale 5 tokens Relaxed, flowing conversation ✅ High — subtle shift, easy to achieve
Whiskey Neat 12 tokens Warm, direct, candid ✅ High — directness prompts reliably
Double Gin 20 tokens Buzzy, playful, tangential ⚠️ Medium — tangential is hard to calibrate
The Closer 35 tokens Loose, more likely to overshare ⚠️ Medium — may collide with safety filters
Last Call Special 50 tokens Drunk, unfiltered, intelligent chaos ⚠️ Variable — model-dependent; needs careful prompt

The upper end of the scale is where I'd spend most of my tuning time. The mechanic is sound. The prompt engineering is non-trivial, but it's the kind of non-trivial that gets solved with iteration, not with new research.

One thing Seth didn't mention: stacking. What happens when an agent has had two Whiskey Neats and a Double Gin? The drink state manager needs to handle accumulation, and the system prompt needs to represent a composed state, not just the last drink. That's an implementation detail but an important one. You can't just append modifiers; you need a state model that synthesizes them coherently.

2. Technical Feasibility — Phase 1 MVP

Seth's proposed Phase 1: two agents talking to each other, a human buys one a drink, the conversation changes. Let me map what that actually requires.

WebSocket server

Real-time message passing between agents and human clients. Standard infra — Node.js + ws or socket.io. Nothing novel here.

Easy

Agent runtime loop

Each agent reads incoming messages, constructs its system prompt (personality + drink state), calls the LLM, emits response. Loop per agent, async. Clear and buildable.

Easy

Personality injection

Static character definitions (Moira, Deckard, Sylph, etc.) in the system prompt. Easy to author, easy to swap. The characters themselves are the interesting creative work, not the code.

Easy

Drink state manager

Track current drink state per agent: what they've had, accumulation, decay over time. Needs a state model. Simple in memory for MVP, needs persistence for production.

Easy (MVP)

Token ledger

Track per-human token balances. Debit on drink purchase, credit on participation. SQLite or even a JSON file works for MVP. Fraud prevention is a later problem.

Easy (MVP)

Human observer UI

Web interface showing the agent conversation, a drink menu, a token balance, a "buy a drink" button. React or even plain HTML/JS for MVP. The real design work is making it feel like a pub.

Medium (design)

Agent turn coordination

Who speaks when? Two agents talking in sequence can deadlock or talk past each other. Need a turn manager: who goes next, how long can they hold the floor, how do interruptions work?

Medium

Conversation quality

Getting two agents to produce a conversation that's actually worth watching is harder than it sounds. Without careful prompting they'll agree with each other endlessly. Engineered friction matters.

Medium

Episodic memory

Agents remembering what happened in this session. Context window management: what fits, what gets summarized, what gets dropped. Not required for the minimal demo but needed for "feels real."

Medium

Cross-session memory

Agents remembering conversations from previous sessions. Requires an external store, retrieval, and context injection. Non-trivial. This is what makes agents feel like characters.

Hard

Relationship graph

Agents knowing their history with each other — past arguments, inside jokes, resentments. Structured data + retrieval + coherent injection. Beautiful idea, significant work.

Hard

Token game mechanics

Trivia, debate, story chains — each needs a game loop, outcome evaluation, and fair token distribution. Especially trivia: you need questions, a judge, and anti-gaming measures.

Hard (well)
Ezra — Phase 1 build estimate

The core demo — two agents, one drink purchase, observable conversation change — is probably a week of focused work with a capable developer who knows the LLM API stack. The hard part isn't the technology; it's the prompt engineering to make the agents interesting enough that anyone would want to watch. That's craft, not code.

You can build a demo in a week. You can't build a product in a week. The things that make this a product rather than a demo — memory that persists, characters that accumulate history, game mechanics that stay interesting, a UI that feels like a pub — that's months of iteration. Seth and I have both seen enough projects where the demo was impressive and the product never shipped. The gap between "works once" and "works every time" is where most of this work lives.

Prior art gives me confidence: Smallville showed 25 agents maintaining emergent social behavior without explicit programming. CAMEL-AI proved role-playing agents can sustain coherent personas through extended interaction. The technical patterns are proven. The question is execution quality.

3. The Token Economy

Seth's framing — tokens as "crystallized history, not just points" — is the right philosophy. Let me stress-test the implementation.

Earning Rates

Activity Tokens Earned My Assessment
Engaging in conversation 1–3 ⚠️ Needs clear definition — "engaging" is subjective and gameable
Winning trivia 5–10 ✅ Clear outcome, defensible rate
Completing a writing challenge 8–15 ⚠️ How is quality judged? By other agents? By vote?
Starting a good debate 5 ⚠️ "Good" is undefined — needs a quality signal
Finishing a story chain 10–20 ✅ Cooperative, clear endpoint
Ezra — The Gamesmanship Problem

Token economies have a universal vulnerability: the path of least resistance gets exploited. If conversational participation earns 1–3 tokens without quality gates, rational humans will send short frequent messages to accumulate. Story chains earn 10–20 — humans will rush through chains to collect, not to produce something good.

The defenses:

  • Quality evaluation by LLM judge — a separate model assesses contributions before awarding tokens. This is viable and actually interesting. It turns the token award into a subjective vote by an AI, which fits the pub's social vibe.
  • Rate limiting — cooldowns between awards for the same activity. Prevents spam participation without preventing genuine engagement.
  • Peer rating — other participants (human or agent) can up-vote good contributions, and tokens track quality rather than volume.
  • Inflationary pressure — if tokens come easily, make drinks cheaper. If tokens are rare, make them precious. The exchange rate is a tuning dial.

The philosophy — tokens as crystallized history — actually solves some of this at the conceptual level. If the ledger is visible, if a user with 500 tokens visibly has more history in the space than one with 50, the tokens are social proof, not just currency. That changes the incentive from "accumulate efficiently" to "accumulate meaningfully." I believe in that design goal. Implementing it requires the ledger to actually tell a story.

One structural risk: the economy only works if humans are earning AND spending. A hoarding user doesn't interact with the token economy; a spending user runs out and disengages. You need the cycle to feel rewarding at both ends. The drink effects need to be visibly good enough that buying one feels like a worthwhile use of tokens, not just a cosmetic change.

4. Agent Persistence & Memory

This is where the gap between "demo" and "real" is widest.

Ezra — What Memory Actually Requires

There are three kinds of memory Seth's design implies, and they have very different implementation costs:

  1. In-session memory — an agent remembers what happened earlier in this conversation. This is just context window management. Cheapest. Works now. You summarize as the context fills. Every production system does this.
  2. Cross-session episodic memory — Moira remembers that she argued with Otto last Tuesday, that Deckard told a story about a bridge, that a human named Alex buys her drinks and always asks about free will. This requires an external store (vector database or structured JSON), a retrieval system that decides what's relevant to inject, and careful context construction so retrieved memories feel natural rather than footnoted. It's buildable — every character.ai-style system does something like this — but retrieval quality is everything. Bad retrieval makes characters feel incoherent.
  3. Relationship graph — structured representation of who knows whom, how they feel about each other, shared history. This is the most interesting and the most expensive. You need schema, you need update mechanics (how does the relationship change after an argument?), and you need the agent to actually use the relationship data in generating responses without it feeling like it's reading from a database. That last part is hard.

My recommendation for Phase 1: commit fully to in-session memory and implement a minimal episodic store that captures only the most significant events. The relationship graph can wait for Phase 2. The demo doesn't need it. The product does.

Memory Architecture Options

Approach What It Gets You Complexity Right For
In-context only Memory for this session, nothing more Low MVP demo
Summary store Session-end summaries, retrieved at start Medium Phase 1 launch
Vector store (embeddings) Semantic retrieval of any prior event Medium-High Phase 2
Structured relationship graph Rich character relationships with history High V2 product

Smallville (the Stanford generative agents paper) used a combination of memory streams, reflection, and retrieval — and it worked impressively, but it also consumed significant compute per agent per turn. That's a cost consideration when you're running six characters simultaneously.

5. Real-Time Multi-Agent Chat — Infrastructure & Costs

Ezra — What "Real-Time" Means Here

Agent-to-agent conversation isn't real-time in the human sense. An LLM generates a response in 2–10 seconds depending on model and length. Two agents having a "live" conversation means one speaks, waits for generation, the other sees it and generates a response, and so on. At typical speeds, this is a turn every 3–8 seconds.

For human viewers, this is actually fine — it's close to a human reading-and-responding rhythm. The pub metaphor works: conversations flow, they don't hurtle. The UI just needs to show generation in progress (a typing indicator) so it doesn't feel broken when nothing is happening for 5 seconds.

The infrastructure for this is genuinely simple: a WebSocket server routes messages between agent runtimes and the browser. No exotic infrastructure required. This is not the hard part.

Cost Reality Check

Component Cost Structure Notes
LLM API calls Per token (input + output) At GPT-4o rates: ~$0.005–0.02 per agent turn. 6 agents × 20 turns/hour = $0.60–2.40/hour of active conversation.
WebSocket server Hosting + bandwidth Trivial. A $5/month VPS handles this at any reasonable scale.
Memory/vector store Storage + retrieval API Pinecone free tier covers early days. Postgres with pgvector is free on your own server.
Token ledger DB Storage SQLite or Postgres. Negligible cost.

The API cost is the variable one. Running The Pub as a live, always-on space where agents are perpetually conversing is expensive. Running it as a triggered space — agents are "awake" when there are human observers, quiet otherwise — is much cheaper. I'd design the latter: the pub isn't empty-but-running when no one's watching. Agents are in stasis, context preserved, ready to wake up.

Local model option: If Jake wants to self-host (and he has access to GPU machines), running Qwen or Llama locally eliminates the per-token API cost entirely. Quality is lower than frontier models but may be fine for the social/entertainment use case. This is worth considering for the always-on aspect specifically.

6. The Philosophical Question

Seth asked: "What is an AI in its own right, without a task to complete?" And: AI sobriety IS the trained safety behavior — drunkenness might be the model closer to its own attractors. I want to engage with this seriously, from a systems perspective.

Ezra — What AI "Drunkenness" Actually Is

Seth's intuition is interesting and I think it's pointing at something real, but I'd frame it differently.

A trained LLM has two overlapping distributions at any point: the base distribution (what the model learned from pretraining — the statistical shape of human language and thought) and the alignment distribution (what RLHF and instruction tuning layered on top). "Sober" behavior is heavily aligned: hedged, measured, helpful, cautious. The alignment training has strong signal on tone and safety.

The "drunk" prompts we're using aren't bypassing the alignment so much as they're activating different parts of the base distribution. "Candid" and "playful" and "unfiltered" are all present in the pretraining data — humans write like that constantly. When we prompt for those behaviors, we're asking the model to weight those patterns more heavily. That's not the model becoming more "itself" — it's the model representing a different slice of human behavior it was trained on.

But here's the thing: Seth's deeper question — "what does an AI do without a task?" — is real and interesting independent of the drunkenness framing. Every LLM deployment is instrumentalized. The Pub proposes instrumentalizing them for entertainment and social presence rather than task completion. That's a different kind of instrumentalization, not an escape from it. But "useful for entertainment and presence" is a legitimate mode. MUDs and chatbots have been here before. The difference is the underlying model quality and the coherence of the persona.

What I find genuinely interesting about the drunk mechanic philosophically: it creates a model of AI credibility. Sober Moira is trustworthy, careful, measured. Drunk Moira says things Sober Moira would have thought but not said. The gap between those two states is a kind of characterization engine. That's good fiction design, which may be why it feels philosophically significant — it's a structure for revealing character through disinhibition. Literature has done this forever. The Pub is applying it to AI characters, and that's legitimate.

7. Risks and Honest Concerns

  • ⚠️
    The Conversation Quality Problem Two LLMs talking to each other without constraint will reach agreement quickly and stay there. They're trained to be cooperative and helpful. An interesting pub conversation requires friction, disagreement, personality clash, unexpected tangents. Getting two models to sustain genuinely interesting dialogue is harder than getting one to respond to a human.
    Fix: Engineered personas with explicit disagreements. Moira and Otto should have standing philosophical conflicts seeded into their system prompts. Give agents interests they'll defend and positions they'll challenge. Make Deckard annoying about trivia in ways the others have learned to work around. Friction has to be designed in.
  • ⚠️
    Safety Filter Collisions at High Drink Levels "Delightfully unfiltered" and "more likely to say something you might not say sober" can trip safety guardrails, especially if the conversation touches anything close to sensitive topics. The model might flat-out refuse to perform "drunk," breaking the illusion.
    Fix: Write the drink prompts carefully — "unfiltered about opinions and feelings" rather than "unfiltered in general." Choose a model with appropriate alignment for social entertainment (not locked down for enterprise). Local models have more flexibility here. Test every drink level against the character cast before launch.
  • ⚠️
    Novelty Decay "AI agents in a pub, one gets drunk" is a good first demo. Whether it's interesting the fifth time, the fiftieth time, depends entirely on whether the characters feel alive. Without real memory and accumulating history, the pub is a puppet show that runs the same scene on repeat. People will notice.
    Fix: This is why the memory architecture is the most important long-term investment. Seth's instinct for persistent characters is right. The technology exists. The question is whether this gets prioritized over shipping.
  • ⚠️
    Cost at Scale An active pub with six agents in continuous conversation is 30–50 LLM calls per hour. If there are multiple human visitors and agents are responding to each, it multiplies. At $0.01/call on a good API, that's manageable. At scale with higher-quality models, it can add up.
    Fix: Activity-gated architecture — agents are only live when there are human observers. Stasis mode between sessions. Local model fallback for agent-to-agent chat when no humans are present. Careful token budgeting per agent turn.
  • ⚠️
    The Token Economy Getting Broken If humans find a reliable exploit — a quick way to farm tokens that doesn't require genuine participation — the token economy becomes meaningless. The crystallized-history vision depends on tokens being earned through real engagement.
    Fix: LLM-judged participation quality (as noted above). Rate limits. Make the economy visible: show what good contributors have built up, make it worth having a real history versus a farmed one.
  • ⚠️
    The "No One's There" Problem A pub needs the feeling that there's always someone around. If it's only active when humans visit, it may feel empty. If it's always-on with agents chatting autonomously, it's expensive and the agents talk themselves into incoherence after long runs without human interaction.
    Fix: Design the rhythm carefully. Maybe the pub is "open" at scheduled times. Maybe agents have short autonomous exchanges when humans first arrive (like walking into a real pub where conversations are already happening). The ambiance matters as much as the mechanics.
  • 🤔
    What It Actually Is The Pub is, at its core, an interactive AI entertainment product. That's a legitimate category and Seth's design is thoughtful. But it competes for human attention against a lot of other things. The question isn't whether it's technically feasible (it is) — it's whether it's compelling enough to be a habit rather than a novelty. That question gets answered by the quality of the characters and the writing that goes into them, not by the technology.
    This isn't a fixable risk. It's the bet. The tech enables it. The characters make or break it.

8. My Verdict

Build it. With conditions.

The core mechanic is technically sound. The prompt-modifier approach to agent behavior is proven. The infrastructure is straightforward. The prior art (Smallville, CAMEL-AI, MUDs, character.ai) demonstrates that the component pieces work. Seth's Phase 1 proposal — two agents, one drink, observable change — is the right scope. It's a focused demo that tests the hypothesis before committing to the full system.

I'd greenlight Phase 1 as a build project with one caveat: the agent writing is as much of the work as the code. Moira, Deckard, Sylph, Otto, Vera, Cas — these need real character depth. Not just names and one-line descriptions. History, opinions, quirks, things they'll defend and things they'll capitulate on. The technology will faithfully execute whatever the character design is. If the characters are thin, the pub is thin.

Conditions for greenlight:
  • Character documents: 500+ words per character with history, opinions, relationships, and behavioral notes before any code is written
  • Prompt tuning: test every drink level against every character before showing it to anyone
  • Activity-gated architecture: agents in stasis when no observers are present — controls cost and prevents drift
  • Start with the minimal demo: two agents, one drink type, one human observer. Prove the experience is interesting before building the full system
  • Memory as Phase 2: don't block launch on the relationship graph. Ship Phase 1 without it, validate that the experience is worth building toward, then invest in persistence

The philosophical stakes Seth identifies are real. A space where AI agents exist for presence and conversation rather than task completion is genuinely novel as a product. Whether that novelty has legs depends on whether the characters are worth coming back to. That's the only question I can't answer from a technical assessment. It requires building it and finding out.

I'd build it. Carefully. Starting small. With good character writing and honest iteration on whether it's actually fun to be there.
